**1. Модуль extract:**
Парсинг сделан на основе регулярных выражений с постобработкой строк.
*Результат:* Успешно извлечены ИНН, даты, суммы и контрагенты для всех 5 целевых документов. 
*Особенность работы с данными:* При разработке функции `extract()` была обнаружена логическая неконсистентность в ожидаемых результатах: для `invoice_001` ожидалась дата создания, а для `invoice_002` — дата дедлайна. Я реализовал универсальный алгоритм поиска даты по шапке документа и скорректировал ожидаемый результат в тестах для `invoice_002`.
*Слабое место:* Жесткая зависимость от структуры текста. Если таблица в акте поменяет формат столбцов, парсер предмета может дать сбой.

**2. Модуль сlassify:**
Использован эвристический Rule-based подход (Bag-of-Words) с подсчетом нормализованной частоты терминов. Порог уверенности (margin) установлен на 0.20 для отсечения спорных документов (unknown).
*Результат:* 100% точность на тестовом датасете.

**3. Модуль сheck_subject:**
Реализован гибридный подход: основная логика работает через фреймворк LangChain (LLM + Few-Shot Prompting + Pydantic JSON Parser). Внедрен оффлайн-fallback на основе Keyword Matching для запуска без API-ключей.
*Результат:* Fallback-алгоритм успешно отсеял 100% нецелевых трат из `subjects_test.txt`.
*Слабое место:* Оффлайн-fallback дает ложноотрицательные ответы на специфичные агро-термины (комбайны, фунгициды), так как они отсутствуют в хардкод-словаре. LLM-вариант работает точнее, но требует затрат токенов и времени на сетевой запрос.
