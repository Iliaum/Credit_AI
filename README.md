# AI-агент для автоматизации проверки целевого использования кредитов

Данный проект представляет собой AI-агент для автоматизации обработки документов (договоров, УПД, счетов), планируемых к заявлению в банк для оплаты льготных кредитов. Проект решает проблему медленной и неточной ручной проверки документов специалистами казначейства.

## 1. Как запустить проект

Убедитесь, что у вас установлен Python 3.11+.

```bash
# 1. Установка зависимостей
pip install -r requirements.txt

# 2. Запуск полного набора тестов
PYTHONPATH=. pytest -v

# 3. Запуск проверки тестовых предметов оплаты (subjects_test.txt)
PYTHONPATH=. python run_subject.py


2. Используемые технологии и обоснование
Python 3.11+: Выбран как стандарт индустрии для Data Science и Backend-разработки.  
PDF

LangChain: Использован для построения LLM-цепочек. Это дает гибкость: можно легко заменить OpenAI на GigaChat или локальную модель без переписывания логики.  
PDF

RegEx (re): Выбраны для модуля экстракции (функция extract). Это обеспечивает мгновенную скорость работы и отсутствие зависимостей от тяжелых нейросетевых моделей на этапе парсинга.  
PDF

Pytest: Использован для покрытия кода тестами, что гарантирует надежность при внесении изменений в пайплайн.  
PDF

3. Архитектура решения
Структура проекта
Plaintext
Project/
├── dataset/             # Тестовые документы (включая намеренно сложный scan_ocr_001.txt)
├── src/                 # Основная логика: extractor, classifier, check_subject
├── tests/               # Unit-тесты
├── requirements.txt     # Зависимости
├── run_subject.py       # CLI-утилита для генерации отчетов
└── README.md            # Документация
Пайплайн обработки
Extract: Функция extract(text) использует иерархические регулярные выражения для поиска ИНН, суммы, даты и предмета.  
PDF

Classify: Эвристический классификатор определяет тип документа (contract, act, invoice, spec) с расчетом confidence-score.  
PDF

Check Subject: Главный узел. Если задан API-ключ, использует LLM (LangChain). Если ключ отсутствует — активируется check_subject_fallback (Keyword matching) для работы без внешних API.  
PDF

4. Компромиссы и допущения
Обработка "неуверенности": В файле scan_ocr_001.txt, который намеренно является сложным для обработки, система не пытается "угадать" данные. Функция extract() честно возвращает None для полей, которые не удалось надежно распознать[cite: 1].  
TXT

RegEx vs NER: Мы отказались от тяжелых NER-моделей в пользу RegEx для достижения высокой скорости обработки на CPU.

Fallback-логика: Для оффлайн-проверки (check_subject) мы ограничились списком ключевых слов. Это гарантирует отсутствие ложноположительных срабатываний, но требует периодического обновления словаря для расширения охвата (например, добавления новых типов техники).

5. Как проверить работу сервиса
Unit-тесты: Запуск pytest проверяет соответствие результатов извлечения эталонным значениям из dataset/ и корректность логики классификации.  
PDF

Валидация отчета: Скрипт run_subject.py считывает файл subjects_test.txt и выводит таблицу результатов (PASS/FAIL) с указанием уверенности (confidence).  
PDF

6. Примеры вызовов
Извлечение данных
Python
from src.extractor import extract
print(extract("Сумма: 1 250 000,00 руб. ИНН 7701234567"))
# {'amount': 1250000.0, 'inn': '7701234567', ...}
Проверка целевого назначения
Python
from src.check_subject import check_subject
matches, conf, reason = check_subject("Поставка семян подсолнечника")
print(f"Match: {matches}, Conf: {conf}, Reason: {reason}")
# Match: True, Conf: 0.85, Reason: [Fallback] Обнаружены маркеры сельскохозяйственной номенклатуры.
